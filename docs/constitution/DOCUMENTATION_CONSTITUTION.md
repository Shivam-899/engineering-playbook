# Documentation Constitution

> Version: 1.0.0
>
> Status: Stable
>
> Last Updated: 2026-06-30

---

# Purpose

Documentation is a product.

It is not an optional deliverable.

Every project should be understandable without relying on tribal knowledge or verbal explanations.

---

# Core Principles

Documentation should be:

- accurate
- concise
- discoverable
- maintainable
- version controlled
- audience focused

---

# Documentation First

Important engineering decisions should be documented when they are made.

Documentation should not be postponed until the end of the project.

---

# Living Documentation

Documentation must evolve together with the codebase.

Outdated documentation is considered a defect.

---

# Single Source of Truth

Each piece of information should have one authoritative location.

Avoid duplicated documentation that can become inconsistent.

---

# Required Documentation

Every production project should include, where applicable:

- README
- Architecture overview
- Setup guide
- Configuration guide
- API documentation
- Deployment guide
- Changelog
- Security notes
- Troubleshooting guide

---

# Audience

Documentation should clearly identify its audience.

Examples include:

- developers
- maintainers
- reviewers
- operators
- end users

---

# Clarity

Prefer:

- simple language
- practical examples
- diagrams where useful
- step-by-step instructions

Avoid unnecessary jargon.

---

# Version Control

Documentation should be reviewed, committed, and versioned alongside the related code changes.

---

# Maintenance

Whenever behavior changes:

- review documentation
- update affected documents
- remove obsolete information

Documentation should never lag behind implementation.

---

# Definition of Done

Work is not complete until:

- required documentation exists
- existing documentation is updated
- examples remain correct
- references remain valid
